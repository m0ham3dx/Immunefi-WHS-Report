The summary of the above diagram is as follows :

  1. Phase 1 - Recon - Which means gathering information about your target, this phase is considered to be the most important phase in this process, poor recon leads to inability to identify bugs.

  2. Phase 2 - Automated Testing - This is often used for baselining. Usually most projects already use automated testing in their testing pipelines. However as the code base gets constantly updated, the probability of logical errors go up, and automated tooling is a fast way (albeit not the most precise way, due to the presence of false positives).

  3. Phase 3 - Exploitable Functions - This step involves functions that are prone to being exploited , some eg: of arbitray functions -

    1. depositFunds() - Crypto IN
    2. withdrawFunds() - Crypto OUT

Hint

üëâ Smart Contract Weakness Classification Registry (SCWCR) - A repository of known vulnerability issues in Ethereum - Not updated regularly.

  1. Phase 4 - Threat Modelling - Once vulnerable code has been identified, the next step is to assess the risk of those functions in simulated conditions. As a good starting point, it was recommended to read about past attacks via bug reports such as those listed in the Immunefi Blog, that give technical details on how the exploit occurred.

Info

üëâ The three core levels of thinking for a Bug Hunter - Excellent tweet thread on threat modelling methodology.

  1. Phase 5 - Fuzzing - Accomplished by tooling that can rapidly input a wide range of inputs to assess the behavior of a function. Note - This step is not necessarily agreed upon, but in my opinion is a fast way to assess the risk of a bug which has been identified.

  2. Phase 6 - POC - A critical step in making the project understand the impact of the finding. This step is often overlooked either due to negligence or lack of technical ability, to write a great POC. The latter issue can be rectified by learning how existing POC‚Äôs for findings have been written. This step is crucial in determining a payout.

Map Core Camponents

  1. Language Upskilling - The most commonly used development languages are :

    1. Solidity- Target -> EVM - Vast majority of blockchain projects or DAPPS are on the Ethereum Chain.

    2. Rust -> Solana/Near - Rust in addition to being a popular language for general development, since it is the closest to the C language, is also used for blockchain apps development.

    3. Javascript - Solidity is close to javascript and tests written for solidity dapps are in javascript which are used by the Hardhat - Popular Development Environment for Dapp Development. However at the time of writing this report, there is a much faster Solidity Development Environment which has been written entirely in rust from the ground up. Its name is Foundry, one big advantage of this tool is that all the tests are written in solidity

    4. Pythonic Languages - Vyper , Brownie & Apeworks are all pythonic languages, which are used for dapp development and testing. Perfect for all the pythonista‚Äôs.
